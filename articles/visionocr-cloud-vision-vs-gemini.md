## はじめに

私は画面のスクリーンショットや紙の研修資料のPDFファイルを文字起こしする際に、自作のOCRツール **VisionOCR** を日常的に使っています。

マルチモーダルLLMの **Gemini** でPDFファイルのOCRを試みましたがあまり思ったよりもうまくいかず、結果として **Google Cloud Vision API** を使うことにしました。

この記事は、LLMのOCRを試したけれど思ったような精度が出ずに悩んでいる方、あるいはこれからOCRツールを選ぼうとしている方に向けて、その判断過程と気づきを共有しようと思います。

https://github.com/yokamoto5742/VisionOCR

## 結論

長文の業務文書OCRには **Google Cloud Vision API** が向いています。LLMによるOCRは、1つの画像ファイルや表・レイアウト保持には強いものの、複数ページのPDFファイルでは **「文章がまるごと抜け落ちる」** と現象が起きました。

LLMは使い勝手がよく万能に見えますが、用途によっては従来のGoogle Cloud Vision APIのほうが確実に成果が上がります。

## 何が起きたか — Gemini で長文が消える現象

衝撃を受けたのは、**10ページ程度のPDFを文字起こしにかけたら、中間の数ページがそのまま抜け落ちた** ことです。

- 「文字が別の単語に置き換わる」というよりは **「文章ががっさり抜け落ちる」**
- 抜けた箇所に欠落を示すマーカーや断りは特になく、出力だけ見ると一見"全ページのOCRが成功している"ように見える
- 同じPDFを Cloud Vision API に通すと、全ページ取得できた

「もっともらしい別の言葉に置き換わる」よりも、**OCRできた部分は正しいが一部欠落しているある方がわかりにくい**と感じました。

## なぜこうなるのか

- **Cloud Vision API**: 画像から「文字として認識できる部分」を機械的に検出・抽出する専用OCR API。文字領域の検出とテキスト変換に特化しており、意味解釈や文脈補完は行わず検出した文字をそのまま出力します。ただし、手書き・低解像度・特殊フォントなど認識困難な文字は欠落することがあります。
- 
- **Gemini**: 画像全体を統計的にパターン処理してテキストを生成するマルチモーダルLLMです。文脈を踏まえた整形や構造化が得意な一方、出力の再現性が保証されず、プロンプト次第では要約・省略・言い換え・補完が発生し得ます。これはLLMが「入力を忠実に転写する」のではなく「学習パターンに基づいて適切な応答を確率的に生成する」ようにトレーニングされているためです。これは他のマルチモーダルLLMでも同じことが言えます。

Cloud Vision APIは検出した文字をそのまま返しますが、Gemini は確率的なテキスト生成を行うため、入力が長くなるほど「全文を逐語的に出力する」よりも「要点をまとめて返す」挙動が出やすくなります。完全転写が必要な用途には、OCR専用APIの方が適していると思われます。

## 比較表

| 観点 | Cloud Vision API | Gemini |
| --- | --- | --- |
| 長文の網羅性 | ◎ 抜けない | △ 抜けることがある |
| 手書き・半角カナ | ◎ | ○ |
| 表・レイアウト保持 | △ バラバラの行になりがち | ◎ Markdown化が得意 |
| 翻訳・整形の同時実行 | × | ◎ プロンプトで指示可能 |
| ハルシネーション | なし | あり |
| レスポンスの安定性 | ◎ 構造が固定 | △ プロンプト調整が必要 |

## どちらを選ぶか

:::message
**OCR結果の欠落や改変がある程度許容されるかかどうか** が選択基準の1つと思います。
:::

- 1文字も漏らせない業務文書・契約書・研修資料 → **Cloud Vision API**
- 表をMarkdown化したい / 翻訳も同時にしたい / 多少の改変は許容できる → **Gemini などのLLM**

私の用途は前者の場合が多く、基本的には Cloud Vision API を利用しています。

## VisionOCR について

上記の判断を踏まえて運用している自作ツールが **VisionOCR** です。

### 主な機能

- 画面の任意の領域をマウスドラッグで選択してOCR実行
- PDFファイルのテキスト抽出（複数ページ対応）
- 日本語の手書き文字・半角カタカナに対応
- テキスト編集機能(読点・句点・改行・空白の除去)
- 追記/上書きモードの切り替え
- OCR結果のクリップボードコピー・ファイル保存

### 動作環境

- Windows
- Python 3.13以上
- Google Cloud Vision API のサービスアカウント認証情報

詳細なインストール手順は README を参照してください。

https://github.com/yokamoto5742/VisionOCR

## おわりに

LLMは便利で、表のMarkdown化や翻訳との同時実行など、専用APIにはできない芸当が確かにあります。一方で、**長文を正確に書き起こす**という一見シンプルな仕事では、OCR専用APIに軍配が上がる場面がまだあります。

「新しい技術が古い技術を全て置き換える」と考えがちですが、**用途に合わせて使い分けるほうが成果が上がる**というのが、今回の検証で得た一番の学びでした。

LLM OCRで精度が出ずに困っている方は、一度 Cloud Vision API ベースのアプローチに戻ってみると、案外あっさり解決するかもしれません。
