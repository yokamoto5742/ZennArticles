## はじめに

業務文書や紙の研修資料のPDFを文字起こしする業務で、自作のOCRツール **VisionOCR** を日常的に使っています。

最近、流行りのマルチモーダルLLMである **Gemini Flash-Lite** に乗り換えようと検証したのですが、結果として **Google Cloud Vision API に戻る** という判断をしました。

この記事は、LLMのOCRを試したけれど思ったような精度が出ずに悩んでいる方、あるいはこれからOCRツールを選ぼうとしている方に向けて、その判断過程と気づきを共有するものです。

https://github.com/yokamoto5742/VisionOCR

## 結論

長文の業務文書OCRには **Google Cloud Vision API** が向いています。LLMによるOCRは、短文・表・レイアウト保持には強いものの、長文では **「文章がまるごと消える」** という事故が起きました。

LLMは使い勝手が良く万能に見えますが、用途によっては従来の専用APIのほうが確実に成果が上がります。

## 何が起きたか — Gemini Flash-Lite で長文が消えた

検証で一番衝撃だったのは、**10ページ程度のPDFを文字起こしにかけたら、中間の数ページがそのまま抜け落ちた** ことです。

崩れ方の特徴はこうでした。

- 「文字が別の単語に置き換わる」のではなく **「文章ごと消える」**
- 抜けた箇所に欠落を示すマーカーや断りもなく、出力だけ見ると一見"成功している"ように見える
- 同じPDFを Cloud Vision API に通すと、全ページ取得できた

「もっともらしい別の言葉に置き換わる」古典的なハルシネーションよりも、**静かに欠落するほうが業務上は怖い**と感じました。レビューしないと気づけないからです。

## なぜこうなるのか

両者は同じ「画像から文字を出す」処理に見えて、設計思想がまったく違います。

- **Cloud Vision API**: 画像から「文字として認識できる部分」を機械的に1文字ずつ抽出する専用API。AIの解釈が入らないため、要約や省略は発生しません。
- **Gemini Flash-Lite**: 画像を"理解"してテキストを生成するマルチモーダルLLM。文脈を踏まえた整形ができる一方、長文では **端折り・要約・省略** が起こり得ます。

LLMは「全部出す」より「うまくまとめる」方向にチューニングされているため、入力が長くなるほど省略のリスクが上がる、というのが実感です。

## 比較表

| 観点 | Cloud Vision API | Gemini Flash-Lite |
| --- | --- | --- |
| 長文の網羅性 | ◎ 抜けない | △ 抜けることがある |
| 手書き・半角カナ | ◎ | ○ |
| 表・レイアウト保持 | △ バラバラの行になりがち | ◎ Markdown化が得意 |
| 翻訳・整形の同時実行 | × | ◎ プロンプトで指示可能 |
| ハルシネーション | なし | あり |
| レスポンスの安定性 | ◎ 構造が固定 | △ プロンプト調整が必要 |

## どう選ぶか

迷ったときに見るべき分岐点は、ほぼ一つだと思っています。

:::message
**ハルシネーション（欠落・改変）が許されない業務かどうか。**
:::

- 1文字も漏らせない業務文書・契約書・研修資料 → **Cloud Vision API**
- 表をMarkdown化したい / 翻訳も同時にしたい / 多少の改変は許容できる → **Gemini などのLLM**

私の用途は前者（紙の研修資料の文字起こし）だったため、Cloud Vision API に戻りました。

## VisionOCR について

上記の判断を踏まえて運用している自作ツールが **VisionOCR** です。

### 主な機能

- 画面の任意の領域をマウスドラッグで選択してOCR実行
- PDFファイルのテキスト抽出（複数ページ対応）
- 日本語の手書き文字・半角カタカナに対応
- テキスト編集機能(読点・句点・改行・空白の除去)
- 追記/上書きモードの切り替え
- OCR結果のクリップボードコピー・ファイル保存

### 動作環境

- Windows
- Python 3.13以上
- Google Cloud Vision API のサービスアカウント認証情報

詳細なインストール手順は README を参照してください。

https://github.com/yokamoto5742/VisionOCR

## おわりに

LLMは便利で、表のMarkdown化や翻訳との同時実行など、専用APIにはできない芸当が確かにあります。一方で、**長文を正確に書き起こす**という一見シンプルな仕事では、OCR専用APIに軍配が上がる場面がまだあります。

「新しい技術が古い技術を全て置き換える」と考えがちですが、**用途に合わせて使い分けるほうが成果が上がる**というのが、今回の検証で得た一番の学びでした。

LLM OCRで精度が出ずに困っている方は、一度 Cloud Vision API ベースのアプローチに戻ってみると、案外あっさり解決するかもしれません。
