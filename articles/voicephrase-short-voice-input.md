## はじめに: 「タイピングより速いの?」への答えは "No, でも"

音声入力ツールを業務に組み込みたいと相談すると、必ずこう聞かれます。

> 「それ、タイピングより本当に速いの?」

正直に答えると **「No」** です。少なくとも、私の手元のタイピング速度は音声入力に負けません。

それでも、私は自作した音声入力ツール **VoicePhrase** を **1日200回以上** 使っています。手放せなくなった理由は速度ではありませんでした。最も予期しなかった変化は、**返信文章の丁寧さと長さが変わったこと** です。

この記事は「音声入力 = 速さの道具」という先入観を持つエンジニアに向けて、**“認知負荷を下げて文章品質を上げる道具”** としての音声入力を提案し、自作ツールの設計判断とハマりどころを共有するものです。

:::message
本記事で紹介する VoicePhrase は Windows 向けの OSS です。Google Cloud Speech-to-Text API を利用するため、利用には Google Cloud のアカウントが必要です。
:::

## なぜ自作したか — メール・チャット返信で限界を感じた瞬間

私の業務は、1日にメールやチャット返信を **200回以上** こなす事務作業が中心です。1回あたりの文章は短く、思考時間より「打つ動作」のコストが支配的になります。

既存の音声入力ツールを試しましたが、どれも事務作業の現場には合いませんでした。

- **Windows 標準の音声入力は日本語認識精度が弱い** — 業務用語や固有名詞で頻繁に誤認識し、結局手で打ち直すことになる
- **ファイル名変更欄やダイアログに貼り付けられない** — 「テキスト入力できる場所」が限定される
- **クラウド型アプリはネット瞬断で音声が消える** — 重要な議事録の最中にネットが切れ、発声し直しになった経験

これらを解決するには、**自分が必要とする最小機能だけを組み合わせた専用ツール** を作るのが結局早い、という結論になりました。

設計の核は次の3点です。

1. **Google Cloud Speech-to-Text API** で日本語認識精度を確保
2. **Win32 SendInput** で貼り付け先を選ばない直接入力
3. **ローカル WAV 保存** で通信瞬断に耐え、F8 キーで再送可能

## 使ってみる — クイックスタート

まずは動かせる状態まで持っていきます。事前に [uv](https://docs.astral.sh/uv/getting-started/installation/) のインストールが必要です。

### 1. クローンと依存パッケージ

```bash
git clone https://github.com/your-repo/VoicePhrase.git
cd VoicePhrase
uv sync
```

### 2. Google Cloud 認証情報を1行 JSON に変換

`.env` に `GOOGLE_CREDENTIALS_JSON` をセットしますが、サービスアカウントキーの JSON は改行を含むため、付属スクリプトで1行化します。

```bash
python scripts/json_minifier.py
```

ファイル選択ダイアログが開くので、Google Cloud Console からダウンロードした JSON を選択してください。タイムスタンプ付きの出力ファイル(例: `credentials_20240430_123456.json`)が生成されます。

### 3. `.env` を作成

```bash:.env
GOOGLE_PROJECT_ID=my-awesome-app-123456
GOOGLE_LOCATION=asia-northeast1
GOOGLE_CREDENTIALS_JSON={"type":"service_account","project_id":"...",...}
```

### 4. 起動

```bash
python main.py
```

Pause キーを押す → 話す → Pause キーで停止。これだけでアクティブウィンドウへテキストが貼り付けられます。

| キー | 機能 |
|------|------|
| Pause | 録音開始 / 停止 |
| F8 | 直前の音声を再変換(ネット瞬断時の再送) |
| F9 | 句読点機能の有効 / 無効 |
| Esc | アプリケーション終了 |

## 設計判断3つ — 他の人がハマりそうな技術的論点

ここからが本題です。VoicePhrase の開発で時間を溶かした3つの判断を、トレードオフ込みで共有します。

### 判断1: 貼り付け先非依存をどう実現するか — Win32 SendInput 採用

最初はクリップボード経由の貼り付け(`Ctrl+V` 送信)を試しました。実装は単純ですが、次の問題に直面します。

- ユーザーのクリップボード履歴を汚染する
- パスワード入力欄など、`Ctrl+V` が無効化されているフィールドで動かない
- **エクスプローラーのファイル名変更欄** や一部ダイアログで挙動が不安定

VoicePhrase は「思いついたファイル名を音声で入力する」「Slack の検索ボックスにキーワードを入れる」といった用途も狙っていたため、これは致命的でした。

最終的に Win32 API の `SendInput` でキー入力をエミュレートする方式に統一しました。OS レベルで仮想キーストロークを送るため、テキスト入力を受け付ける場所ならほぼどこでも動作します。

:::message
SendInput は IME の状態に依存するため、入力前に IME を一時的に無効化するなどの細かな前処理が必要です。実装は `service/paste_backend` を参照してください。
:::

### 判断2: ネット瞬断耐性 — ローカル WAV 保存 + F8 再送

クラウド型音声入力の最大の弱点は「ネットワーク失敗時に音声が消える」ことです。重要な会議の議事録を取っている最中にこれをやられると、致命的な作業ロスが発生します。

VoicePhrase は録音した音声を **常にローカルの WAV ファイルとして一時保存** してから API に送ります。送信が失敗した場合、ユーザーは **F8 キーを押すだけで直前の音声を再送** できます。

```
[マイク入力] → [WAV保存] → [STT API送信] → [テキスト貼り付け]
                  ↑              ↓ (失敗時)
                  └── F8で再送 ──┘
```

「発声し直さなくていい」という体験は、実際に使うと予想以上に効きます。**ネット環境を気にせず話せる** という心理的安全性が、ツールへの信頼につながりました。

### 判断3: 【目玉】Tkinter + バックグラウンドスレッドの落とし穴 — UIQueueProcessor 導入記

最も時間を溶かしたのがこれです。**Tkinter + 音声処理のアプリを自作する人には、特に共有したい話** です。

#### 初期実装で起きたこと

録音処理・API 送信・結果表示はすべて別スレッドで動かしていました。素朴にこう書いていたのです。

```python
# ❌ アンチパターン: 別スレッドから直接 Tkinter を更新
def on_transcription_done(text: str):
    self.label.config(text=text)  # 別スレッドから呼ばれる
    self.window.update()
```

これで **原因不明のフリーズが頻発** しました。再現条件が掴めず、長時間動かしていると突然 UI が固まる。最初は Google Cloud API のタイムアウトを疑い、リトライ処理やタイムアウト値を何度も変更しました。**まったく改善しません**。

#### 原因: Tkinter はシングルスレッド前提

Tkinter のウィジェット操作はメインスレッド(イベントループを回しているスレッド)からのみ呼ぶ前提で設計されています。別スレッドからの操作は、運が良ければ動き、運が悪ければデッドロックする、という不安定な挙動になります。

#### 解決: すべての UI 更新をキュー経由に統一

`UIQueueProcessor` という小さなクラスを導入し、**UI 更新は必ずキューに積み、メインスレッドの `after()` で逐次取り出して実行する** ように統一しました。

```python:app/ui_queue_processor.py
import queue

class UIQueueProcessor:
    def __init__(self, root):
        self.root = root
        self.queue = queue.Queue()
        self._poll()

    def schedule_callback(self, callback):
        # 別スレッドから安全に呼べる
        self.queue.put(callback)

    def _poll(self):
        try:
            while True:
                callback = self.queue.get_nowait()
                callback()
        except queue.Empty:
            pass
        self.root.after(10, self._poll)
```

呼び出し側はこうなります。

```python
# ✅ 別スレッドから UI を更新する正しい方法
def on_transcription_done(text: str):
    ui_queue.schedule_callback(lambda: self.label.config(text=text))
```

この変更で **フリーズは完全に消滅** しました。Tkinter + バックグラウンドスレッド構成のアプリを書くなら、最初からこのパターンを採用することを強く推奨します。

:::message alert
「UI 更新を別スレッドから呼んでも一見動く」のが最大の罠です。開発中は問題なく見えても、本番運用で突然フリーズします。**全 UI 更新をキュー経由に統一する** というルールを最初に決めてください。
:::

## 認識精度を上げる小技 — 専門用語登録と置換ルール

業務用語や医療用語は、汎用の音声認識モデルでは正しく拾えません。VoicePhrase は2段構えでこれを補正します。

### フレーズヒント(`data/technical_terms.txt`)

Google Cloud Speech-to-Text API の `speech_recognition_hints` に渡す専門用語リストです。1行1フレーズで登録します。

```text:data/technical_terms.txt
$OOV_CLASS_DIGIT_SEQUENCE
$OPERAND
加齢黄斑変性
```

`$OOV_CLASS_*` は Google STT API の **クラストークン** で、数字列や演算子といった「カテゴリ」を認識ヒントとして与えられます。日本語ドキュメントが薄い領域なので、業務で数字や記号の認識精度に困っている方はぜひ試してみてください。

### 後処理置換(`data/replacements.txt`)

それでも誤認識する単語は、CSV 形式で機械的に置換します。

```csv:data/replacements.txt
# 医療系の同音異義語を補正
小児体,硝子体

# 不要な疑問符を句点に
?,。
```

「フレーズヒントで予防 → 置換ルールで治療」の二段構えにすることで、運用しながら少しずつ精度が育っていきます。

## 使い続けてわかった本当の効能

冒頭に戻ります。VoicePhrase を1日200回以上使うようになって、**最も予期しなかった変化** は速度ではありませんでした。

**返信文章の丁寧さと長さが変わったのです。**

タイピングで返信を書いていた頃の私は、無意識に「打つコスト」を節約していました。「了解しました」で済むところを「承知しました、◯◯の件は△△までに対応します」と書く余力が、なかなか出ない。打鍵数を減らす方向に文章が縮こまっていく。

音声入力に切り替えてから、この **「打つコストの節約」というバイアス** が外れました。話す方が文章の自然な長さに近いため、結果として伝わる内容が豊かになります。タイピングしながら考えていた認知負荷が減り、**「相手にどう伝えるか」だけに集中できる** ようになりました。

副産物として、生成 AI へのプロンプト入力も劇的に楽になりました。長文プロンプトを書くハードルが下がり、AI の出力品質が上がる、という好循環が生まれています。
